# OpenAPI 3.1 → 3.0 spec-authoring constraints

`docs/SPEC.md` commits jamsesh to OpenAPI 3.1 contracts.
oapi-codegen mainline v2.7.0 does NOT yet support 3.1 — it's
"awaiting upstream support" (kin-openapi parser). An experimental
fork exists (`oapi-codegen-exp`) but is explicitly NOT
production-ready.

## Workaround

Until 3.1 lands in mainline, author `docs/openapi.yaml` against
the **3.0-compatible subset** of 3.1 features and declare
`openapi: 3.0.3` at the top.

`openapi-typescript` on the client side handles both 3.0 and 3.1
natively — no client-side change needed. The TS types are
identical for the constrained subset.

## Spec-authoring rules

| 3.1 idiom (avoid)                  | 3.0-compatible equivalent          |
|------------------------------------|------------------------------------|
| `webhooks:` top-level              | model as regular operations        |
| `type: [string, "null"]`           | `type: string` + `nullable: true`  |
| `examples:` array on schemas       | single `example:`                  |
| `prefixItems:` (tuple)             | enumerate as `oneOf` or document   |
| `if` / `then` / `else`             | model with `oneOf` + discriminator |
| `dependentRequired`                | combine via `oneOf`                |
| `const:`                           | `enum:` with one value             |
| `contentMediaType` on string       | use `requestBody` content map      |
| `$dynamicRef` / `$dynamicAnchor`   | flatten into named components      |

## Migration trigger

When kin-openapi 3.1 lands in oapi-codegen mainline:

1. Update `oapi-codegen.yaml` version pin.
2. Change `openapi: 3.0.3` → `openapi: 3.1.0` in
   `docs/openapi.yaml`.
3. Lift the constraints above where useful (especially `type:
   ["string", "null"]` for cleaner nullable handling and
   `examples:` arrays for richer docs).
4. Regenerate; run `make ci-generate` to confirm no drift.
5. Update this references file with the new pin.
