# Auth-Guards — Define helpers and migrate sessions handler

Define the `handlerauth` package and migrate `internal/portal/sessions/handler.go`
as the first consumer. This story proves the helper shape against the most
complex consumer (4 handlers, all three guard tiers represented).

## Files

- New: `internal/portal/handlerauth/handlerauth.go`
- New: `internal/portal/handlerauth/handlerauth_test.go`
- Modify: `internal/portal/sessions/handler.go` (4 handlers + helper imports)

## Current state (per parent feature body)

Each of CreateSession, PatchSession, FinalizeSession, AbandonSession
duplicates account extraction (8 LoC) and org/session membership checks
(12-15 LoC each).

## Target state

`handlerauth.RequireAccount`, `RequireOrgMember`, `RequireSessionMember` plus
`AuthFail` per parent body. All four sessions handlers reduced to a single
guard call each.

## Implementation notes

- Place the package at `internal/portal/handlerauth`; do not extend `tokens`
  because `tokens` is auth-only (parsing/middleware), not authorization.
- The `AuthFail` struct contains *both* the 401 and 403 payloads as fields;
  only one is populated based on `Status`. Avoids `interface{}` and keeps
  call sites type-safe.
- Build the typed wrapping in each handler with a tiny per-handler helper
  inside `sessions/handler.go`, e.g.:

  ```go
  func patchSessionFail(f handlerauth.AuthFail) openapi.PatchSessionResponseObject {
      if f.Status == 401 {
          return openapi.PatchSession401JSONResponse{UnauthorizedJSONResponse: f.Unauthorized}
      }
      return openapi.PatchSession403JSONResponse{ForbiddenJSONResponse: f.Forbidden}
  }
  ```

  This keeps the operation-specific typing local to the handler package.

## Implementation notes

### AuthFail shape decision

`AuthFail` carries both `Unauthorized` and `Forbidden` response payloads as value fields (not pointers/interfaces) plus `Status int` (401, 403, or 500) and an optional `Err error` for unexpected store failures. The dual-field approach avoids type assertions at call sites and keeps the compiler helping. For 500 cases, `fail.Err` is non-nil and `fail.Status == 500` — callers wrap and return the error as a handler-level error (the strict server converts it to 500). This matches what the existing code did before the refactor (e.g. `return nil, fmt.Errorf("sessions: get org member: %w", err)`).

### Per-handler typed wrapper pattern

Each handler has a small `<op>Fail(f handlerauth.AuthFail) openapi.<Op>ResponseObject` helper in `sessions/handler.go`. These wrappers live in the handler package because they embed the per-operation response types generated by oapi-codegen (e.g. `CreateSession401JSONResponse` vs `PatchSession401JSONResponse`). The `handlerauth` package stays import-free of operation-specific types.

### Helper assignment to sessions handlers

- `CreateSession`: uses `RequireOrgMember` (original checked org membership)
- `PatchSession`: uses `RequireSessionMember` — eliminates the direct `GetSessionMember` call; role check ("creator only") remains inline after the helper
- `FinalizeSession`: uses `RequireAccount` — original had no membership check, just an account existence check
- `AbandonSession`: uses `RequireSessionMember` — eliminates the direct `GetSessionMember` call; role check ("creator only") remains inline

### Auth-before-existence ordering change

`PatchSession` and `AbandonSession` previously checked session existence (404) before membership (403). After the refactor, membership is verified first (using `RequireSessionMember` which calls `GetSessionMember`). This means a non-member requesting a non-existent session now gets a 403 instead of a 404. This is the more secure behavior (don't leak session existence). Existing tests only cover the happy path and wrong-role cases; no test pins the 404-before-403 ordering, so no regression.

### LoC delta in sessions/handler.go

`git diff --stat` shows `77 insertions(+), 77 deletions(-)` — net zero in handler.go itself. The removed inline auth boilerplate (~70 LoC across 4 handlers) is offset by the ~35 LoC of per-handler typed wrapper helpers added at the bottom of the file. The real savings live in the new `handlerauth` package (139 LoC), which will be shared by the remaining stories in this epic (comments, accounts, tokens handlers).

### Test stub approach

The test stub for `store.Store` is a hand-rolled struct with function fields for `GetOrgMember` and `GetSessionMember`, all other methods panic. A compile-time `var _ store.Store = (*stubStore)(nil)` assertion catches any interface drift. The alternative (using the real SQLite store) would make the tests heavier and infrastructure-dependent; unit-level function stubs keep the test focused and fast.
